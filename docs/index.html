v3.1.0 — Arquitectura de Contextos Dinámicos

Laravel Module Maker

Genera hasta 24 archivos por contexto: controllers, services, repositories, jobs, notifications, console commands, exceptions, migrations, factories, seeders, tests y vistas Vue — con un solo comando artisan, en segundos.

composer require innodite/laravel-module-maker
⚡
Generación Completa
Scaffold de backend PHP + frontend Vue en un único comando. Todos los archivos correctamente nombrados y enlazados.
🎯
Contextos Dinámicos
Define contexts.json con 4 tipos: central, shared, tenant_shared y tenant. Cada uno genera rutas e inyección de prefijos correctos.
🔗
Bridge Frontend
Middleware que inyecta automáticamente auth.context y auth.permissions vía Inertia shared props.
📋
Auditoría
Registro NDJSON de cada archivo generado: comando, contexto, módulo, archivo, timestamp y hash SHA-256.
Primeros Pasos

Instalación

Sigue estos cuatro pasos para tener el paquete completamente configurado en tu aplicación Laravel.

1
Requerir el paquete vía Composer
bash
composer require innodite/laravel-module-maker
2
Ejecutar el setup inicial (publica config y stubs)
bash
php artisan module:setup

Publica contexts.json, stubs personalizables y el archivo de configuración principal.

3
Publicar los composables Vue
bash
php artisan module:publish-frontend

Publica useModuleContext.js y usePermissions.js en resources/js/composables/.

4
Registrar el middleware Bridge en bootstrap/app.php
php
use Innodite\ModuleMaker\Http\Middleware\InnoditeBridgeMiddleware;

->withMiddleware(function (Middleware $middleware) {
    $middleware->appendToGroup('web', InnoditeBridgeMiddleware::class);
    $middleware->appendToGroup('api', InnoditeBridgeMiddleware::class);
})

      "permission_prefix": "shared",
      "route_file": ["web.php", "tenant.php"],
      "middleware": ["web", "auth"],
      "sub_contexts": []

    {
      "name": "tenant_shared",
      "type": "tenant_shared",
      "route_prefix": null,
      "permission_prefix": "tenant",
      "route_file": "tenant.php",
      "middleware": ["web", "auth", "tenant"],
      "sub_contexts": []

    {
      "name": "tenant",
      "type": "tenant",
      "route_prefix": null,
      "permission_prefix": "tenant",
      "route_file": "tenant.php",

Controller generado — RendersInertiaModule

Regla obligatoria: Todos los controllers usan el trait RendersInertiaModule. El método index() siempre retorna renderModule() para navegar a la vista. Los datos se cargan exclusivamente vía axios desde la vista Vue al montarse.

php
class CentralUserController extends Controller
{
    use RendersInertiaModule;

    public function index(): InertiaResponse
    {
        return $this->renderModule('User', 'CentralUserIndex'); // Solo navega a la vista
    }

    public function list(Request $request): JsonResponse
    {
        return response()->json($this->service->paginate($request->all())); // Datos para axios
    }

    public function store(Request $request): JsonResponse
    {
        return response()->json($this->service->create($request->validated()), 201);
    }
}

Ruta inyectada en web.php

php
Route::prefix('central')->middleware(['web', 'auth'])->group(function () {
    Route::resource('users', \App\Http\Controllers\Central\CentralUsersController::class);
});

Cómo contextRoute() resuelve

javascript
const { contextRoute } = useModuleContext()

contextRoute('users.index')  // => "central.users.index"
contextRoute('users.store')  // => "central.users.store"
Contexto

Shared

El contexto shared es especial: genera módulos accesibles tanto desde el panel central como desde el panel tenant. Por eso route_file es un array ["web.php", "tenant.php"] — el paquete inyecta el bloque de rutas en ambos archivos.

Dualidad de rutas: El mismo controller atiende requests de /shared/users (web.php) y de /shared/users dentro del tenant (tenant.php). El middleware Bridge distingue el contexto.

bash
php artisan make:module Users --context=shared

Rutas inyectadas (dos archivos)

php
// routes/web.php
Route::prefix('shared')->middleware(['web', 'auth'])->group(function () {
    Route::resource('users', \App\Http\Controllers\Shared\SharedUsersController::class);
});
php
// routes/tenant.php
Route::prefix('shared')->middleware(['web', 'auth', 'tenant'])->group(function () {
    Route::resource('users', \App\Http\Controllers\Shared\SharedUsersController::class);

Tenant Shared

El contexto tenant_shared genera módulos que viven exclusivamente dentro del scope tenant, compartidos por todos los tenants, pero sin route_prefix. Las rutas van directamente en tenant.php sin prefijo de URL adicional.

Sin route_prefix: route_prefix es null, por lo que contextRoute('users.index') resuelve directamente a tenant.users.index usando el permission_prefix como referencia de nombre.

bash
php artisan make:module Users --context=tenant_shared

Frontend

Bridge Middleware

InnoditeBridgeMiddleware se ejecuta en cada request y comparte vía Inertia::share() los datos de contexto y permisos necesarios para el frontend.

Props inyectadas

PropTipoEjemploDescripción
auth.context.route_prefixstring|null"central"Prefijo de ruta del contexto del usuario autenticado
auth.context.permission_prefixstring"central"Prefijo para construir nombres de permisos
auth.context.namestring"central"Nombre del contexto
auth.permissionsarray["central.users.view"]Lista de permisos del usuario autenticado

Cadena de resolución de permisos

El Bridge intenta resolver permisos en este orden: Spatie Permission → InnoditeUserPermissions (custom) → array vacío (fail-safe). Nunca lanza excepción si el sistema de permisos no está instalado.

php
// Resolución interna del Bridge (simplificado)
try {
    $permissions = $user->getAllPermissions()->pluck('name')->toArray(); // Spatie
} catch (\Throwable $e) {
    try {
        $permissions = $user->innoditePermissions(); // Custom interface
    } catch (\Throwable $e2) {
        $permissions = []; // Fail-safe
    }
}

Inertia::share([
    'auth.context' => [
        'name'               => $context['name'],
        'route_prefix'       => $context['route_prefix'],
        'permission_prefix'  => $context['permission_prefix'],
    ],
    'auth.permissions' => $permissions,
]);

Registro en bootstrap/app.php

php
use Innodite\ModuleMaker\Http\Middleware\InnoditeBridgeMiddleware;

return Application::configure(basePath: dirname(__DIR__))
    ->withRouting(...)
    ->withMiddleware(function (Middleware $middleware) {
        $middleware->appendToGroup('web', InnoditeBridgeMiddleware::class);
        $middleware->appendToGroup('api', InnoditeBridgeMiddleware::class);
    })
    ->create();
Frontend

Composables Vue

El paquete publica dos composables en resources/js/composables/ que consumen las props del Bridge.

useModuleContext.js

Provee contextRoute() para construir nombres de rutas de forma context-aware, y contextPermission() para construir nombres de permisos.

javascript
import { useModuleContext } from '@/composables/useModuleContext'

const { contextRoute, contextPermission, context } = useModuleContext()

// Construcción de rutas:
contextRoute('users.index')    // "central.users.index"
contextRoute('users.store')    // "central.users.store"

// Construcción de permisos:
contextPermission('users.view')  // "central.users.view"
contextPermission('users.edit')  // "central.users.edit"

// Acceso al contexto completo:
console.log(context.value.name)           // "central"
console.log(context.value.route_prefix)   // "central"

usePermissions.js

Provee helpers reactivos para verificar permisos contra la lista inyectada por el Bridge.

javascript
import { usePermissions } from '@/composables/usePermissions'
import { useModuleContext } from '@/composables/useModuleContext'

const { can, canAny, canAll } = usePermissions()
const { contextPermission } = useModuleContext()

// Verificación individual:
can(contextPermission('users.view'))     // true | false

// Al menos uno:
canAny(['central.users.view', 'central.users.edit'])  // true | false

// Todos:
canAll(['central.users.view', 'central.users.edit'])  // true | false

Estrategia de permisos dual

HelperComportamientoCaso de uso
can(perm)Comprueba si perm está en auth.permissionsBotones individuales, rutas protegidas
canAny(perms[])OR — al menos unoMostrar sección si tiene cualquier acceso
canAll(perms[])AND — todos requeridosAcciones destructivas con múltiples permisos

Vistas Vue Generadas

Cada módulo genera 4 vistas Inertia.js. Las vistas usan axios para llamadas API y los composables para routing y permisos.

Arquitectura de vistas

VistaEstrategia de datosDescripción
*Index.vueInertia props + axiosListado con paginación, búsqueda y acciones
*Create.vueInertia form helpersFormulario de creación con validación reactiva
*Edit.vueInertia props + formFormulario de edición prellenado
*Show.vueInertia propsVista de detalle, solo lectura

Ejemplo: CentralUsersIndex.vue

javascript
<script setup>
import { ref, onMounted } from 'vue'
import { useModuleContext } from '@/composables/useModuleContext'
import { usePermissions } from '@/composables/usePermissions'

const { contextRoute } = useModuleContext()
const { can } = usePermissions()
const users = ref([])

onMounted(async () => {
  const { data } = await axios.get(route(contextRoute('users.index')))
  users.value = data.data
})
</script>

<template>
  <div>
    <button
      v-if="can('central.users.create')"
      @click="$inertia.visit(route(contextRoute('users.create')))"
    >
      Nuevo Usuario
    </button>
    <table>
      <tr v-for="user in users" :key="user.id">
        <td>{{ user.name }}</td>
        <td>{{ user.email }}</td>
      </tr>
    </table>
  </div>
</template>

Migración por manifiesto: innodite:migrate-plan

Este comando resuelve coordenadas lógicas y ejecuta cada migración en el orden declarado, evitando errores por dependencias implícitas entre módulos.

Antes de ejecutar, valida la conexión objetivo y verifica que la base de datos exista para evitar procesos parciales o disparados contra una BD incorrecta.

bash
# Manifiesto por defecto
php artisan innodite:migrate-plan

# Manifiesto específico
php artisan innodite:migrate-plan --manifest=tenant_innodite_order.json

# Simulación sin ejecutar en BD
php artisan innodite:migrate-plan --manifest=tenant_innodite_order.json --dry-run

# Ejecutar migraciones y luego seeders
php artisan innodite:migrate-plan --manifest=tenant_innodite_order.json --seed

Ejecución individual de migración: innodite:migrate-one

Permite ejecutar una migración puntual sin correr el manifiesto completo. Antes de ejecutar, muestra exactamente dónde va a correr y puede registrar la coordenada en el manifiesto si todavía no existe.

bash
# Ejecutar una migración específica
php artisan innodite:migrate-one "Products:Tenant/EnergySpain/2026_01_01_000001_create_products_table.php"

# Forzar un manifiesto concreto
php artisan innodite:migrate-one "Forms:Shared/2026_01_01_000001_create_forms_table.php" --manifest=central_order.json

# Simular sin tocar manifiesto ni BD
php artisan innodite:migrate-one "Forms:Shared/2026_01_01_000001_create_forms_table.php" --dry-run

# Omitir confirmaciones interactivas
php artisan innodite:migrate-one "Products:Tenant/EnergySpain/2026_01_01_000001_create_products_table.php" --yes

Sincronización automática: innodite:migration-sync

Escanea todo Modules/*/Database/Migrations y Modules/*/Database/Seeders, calcula coordenadas y agrega faltantes al manifiesto sin duplicar registros.

bash
# Sincronizar manifiesto por defecto
php artisan innodite:migration-sync

# Sincronizar manifiesto específico
php artisan innodite:migration-sync --manifest=tenant_innodite_order.json

# Ver faltantes sin escribir
php artisan innodite:migration-sync --manifest=tenant_innodite_order.json --dry-run

Flujo interno del sync

PasoAcción
1Crea module-maker-config/migrations/ si no existe
2Crea manifiesto vacío si no existe
3Escanea migraciones y seeders contextuales de todos los módulos
4Compara con el contenido actual del manifiesto
5Hace append solo de faltantes (idempotente)

Práctica recomendada: Ejecuta innodite:migration-sync --dry-run en CI para detectar drift entre código y manifiesto antes del deploy.

Publicar Assets

El paquete soporta publicación selectiva de sus componentes mediante tags de vendor publish.

bash
# Publicar todo
php artisan vendor:publish --provider="Innodite\ModuleMaker\ModuleMakerServiceProvider"

# Solo configuración
php artisan vendor:publish --tag=module-maker-config

# Solo stubs (para personalizar plantillas)
php artisan vendor:publish --tag=module-maker-stubs

# Solo composables Vue
php artisan module:publish-frontend

Estructura de stubs contextuales (4 carpetas × 26 stubs)

text
module-maker-config/stubs/contextual/
├── Central/          ← Personaliza stubs solo para contexto Central
│   ├── controller.stub
│   ├── service.stub
│   ├── job.stub
│   └── ... (26 stubs)
├── Shared/           ← Personaliza stubs solo para contexto Shared
├── TenantShared/     ← Personaliza stubs solo para Tenant Shared
└── TenantName/       ← Personaliza stubs para todos los tenants específicos

Resolución por prioridad: El paquete busca el stub en este orden: (1) custom por contexto, (2) custom genérico, (3) paquete por contexto, (4) paquete genérico. Puedes sobrescribir solo el stub de Central sin afectar los demás contextos.
