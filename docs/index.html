Laravel Module Maker

A powerful Artisan-first package for generating and managing modular Laravel applications with first-class multi-tenancy support via explicit context configuration.

composer require innodite/laravel-module-maker
🏗️
Full Module Scaffolding
Generate complete modules with Model, Controller, Service, Repository, Migrations, Seeders, Tests and Vue views in one command.
🔑
Explicit Multi-Tenancy
Define Central, Shared, Tenant-Shared and per-Tenant contexts in a single contexts.json. No magic — you control every connection.
📋
Migration Manifests
Order-file system ensures migrations run in the exact sequence you define, per context, with full rollback and dry-run support.
🛡️
Guard Rail Safety
Connection Guard validates every migration target against config/database.php before touching the database. No silent fallbacks.
➕
Multi-Entity Modules
Add entities to existing modules with innodite:add-entity. Every entity gets its own isolated subfolder: Models/Central/Role/CentralRole.php.
✏️
Custom Stubs
Override any stub per context folder. Your project's stubs always take priority over the package defaults.
🧪
Test-First DX
Built-in test runner with per-module and per-context filtering, HTML/Clover coverage reports, and sync helpers.

The package auto-discovers its service provider in Laravel 12. Just require it and run the setup command.

# 2. Run setup (creates folders + config)
php artisan innodite:module-setup

# 3. Verify your environment

Configuration

Contexts — contexts.json

Every behavior in Module Maker is driven by module-maker-config/contexts.json. This file defines the "worlds" your modules live in.

⚠️ Hybrid Structure Rule

central, shared, and tenant_shared are JSON objects. tenant is a JSON array (supports multiple tenants). Never flatten everything into arrays.

Available Context Types

TypeDescriptionConnection
centralShared admin/back-office database — one instanceRequired (connection_key)
sharedCross-tenant shared resources (no dedicated DB)Not required
tenant_sharedLogic shared across all tenants, using each tenant's DBNot required (resolved per tenant)
tenantIndividual tenant instance with its own DB connectionRequired (connection_key)

Context Fields Reference

FieldTypeDescription
idstring (kebab-case)Unique identifier — used in manifests, CLI args, and route matching
tenancy_strategy"manual" or absentSet to "manual" to enable connection Guard Rail
connection_keystringName of the connection in config/database.php
class_prefixPascalCasePHP class prefix e.g. Central, TenantAcme
folderstringSubfolder within each component directory
namespace_pathstringPHP namespace fragment (uses \\ separator)
route_filestring or arrayRoute file(s): web.php, tenant.php, api.php
route_prefixstringURL prefix for routes
route_namestringNamed route prefix (include trailing dot e.g. central.)
permission_prefixstringSpatie permission prefix
route_middlewarearrayExtra middleware for this context's route group
Configuration

Generated Module Structure

Every module generated by innodite:make-module follows this directory layout. Since v3.5, core components (Model, Controller, Service, Repository, Migration, Request) are placed inside an entity subfolder: {Type}/{Context}/{Entity}/.

Modules/{ModuleName}/ ├── Database/ │ ├── Factories/{Central/,Shared/,Tenant/...}/{Entity}/ │ ├── Migrations/{Central/,Shared/,Tenant/...}/{Entity}/ │ └── Seeders/{Central/,Shared/,Tenant/...}/{Entity}/ ├── Docs/ │ ├── history.md │ ├── architecture.md │ └── schema.md ├── Http/ │ ├── Controllers/{Central/,Shared/,Tenant/...}/{Entity}/ │ ├── Middleware/ │ └── Requests/{Central/,Shared/,Tenant/...}/{Entity}/ ├── Models/{Central/,Shared/,Tenant/...}/{Entity}/ ├── Providers/ │ └── {ModuleName}ServiceProvider.php ├── Repositories/ │ ├── Contracts/{...}/{Entity}/ │ └── {Central/,Shared/,Tenant/...}/{Entity}/ ├── Resources/ │ ├── js/Pages/{Central/,Shared/,Tenant/...} ← Vue components (no entity subfolder) │ ├── views/ │ └── lang/ ├── Routes/ │ ├── web.php │ ├── tenant.php │ └── api.php ├── Services/ │ ├── Contracts/{...}/{Entity}/ │ └── {Central/,Shared/,Tenant/...}/{Entity}/ ├── Console/Commands/{...} ├── Jobs/{...} ├── Notifications/{...} ├── Exceptions/ └── Tests/ ├── Feature/{Central/,Shared/,Tenant/...} ├── Unit/{...} └── Support/
💡 Entity subfolder pattern (v3.5+)

Core components follow Models/Central/User/CentralUser.php — entity name as subfolder, keeping context and naming convention intact. Use innodite:add-entity to add more entities (e.g. Role, Permission) to an existing module under the same pattern.

php artisan innodite:module-setup

Initializes the complete Module Maker workspace. Creates the Modules/ directory, publishes the contexts.json template, copies all stubs, and patches your DatabaseSeeder.php to include the module seeder chain. Run this once per project.

💡 Re-running setup

The command is safe to re-run — it will not overwrite your contexts.json or any custom stubs you've modified. It only creates files that don't already exist.

✨
php artisan innodite:make-module {name} [options]

The core generator. Scaffolds a complete module with all its components (Model, Controller, Service, Repository, Migration, Seeder, Requests, Factory, Tests, Vue views) for a specific context. Since v3.5, each component is placed inside an entity subfolder: Models/Central/User/CentralUser.php. To add more entities to an existing module, use innodite:add-entity.

generator multi-context

Signature

bash
php artisan innodite:make-module {name}
                                {--context=}        # explicit context ID
                                {--json}            # use module-maker-config/{module}.json
                                {--no-routes}       # skip route injection
                                {-M|--model}        # generate only the Model
                                {-C|--controller}   # generate only the Controller
                                {-S|--service}      # generate only the Service
                                {-R|--repository}   # generate only the Repository
                                {-G|--migration}    # generate only the Migration
                                {-Q|--request}      # generate only the Request

Options

OptionTypeDescription
nameargumentEntity name in singular, PascalCase. e.g. User, BlogPost
--context=optionContext ID from contexts.json. e.g. central, acme. If omitted, an interactive selector appears.
--jsonflagLoad component configuration from module-maker-config/{name}.json instead of defaults
--no-routesflagSkip automatic route injection into the route files
-M / --modelflagGenerate only the Model component
-C / --controllerflagGenerate only the Controller component
-S / --serviceflagGenerate only the Service + ServiceInterface
-R / --repositoryflagGenerate only the Repository + RepositoryInterface
-G / --migrationflagGenerate only the Migration file
-Q / --requestflagGenerate only Request classes (Store + Update)

ℹ️ Name Validation

The name argument is validated against PHP/Laravel reserved words and must be a valid identifier (letters, numbers, underscore). It is automatically converted to PascalCase.

Generated Route Injection

When route injection is active (default), the command adds route blocks to the matching route file using idempotent markers. Re-running the command will never duplicate routes.

phproutes/web.php (after injection)
use Modules\User\Http\Controllers\Central\User\CentralUserController;

Route::group(['prefix' => 'central', 'middleware' => ['web']], function () {
    Route::resource('users', CentralUserController::class);
    // {{CENTRAL_ROUTES_END}}
});
📋
php artisan innodite:migrate-plan [options]

Executes all migrations and optionally seeders defined in a manifest order file (*.order.json). Validates the DB connection through the Guard Rail before touching anything. Supports dry-run mode for safe previewing.

migrations seeders

Signature

bash
php artisan innodite:migrate-plan
    {--manifest=}   # manifest file to use (default: central.order.json)
    {--dry-run}     # preview plan without executing
    {--seed}        # also run seeders after migrations

Options

OptionDescription
--manifest=Name or path of the manifest file. Defaults to central.order.json. Can be just the filename (e.g. acme.order.json).
--dry-runShows the migration plan without executing anything. DB connection check is skipped in dry-run.
--seedAfter all migrations complete, run the seeders listed in the manifest.

Examples

Example 1 — Run central migrations
bash
# Default manifest = central.order.json
php artisan innodite:migrate-plan

# Explicit manifest
php artisan innodite:migrate-plan --manifest=central.order.json
Example 2 — Run migrations for a specific tenant
bash
# Tenant "acme" (must have connection_key "acme_db" in config/database.php)
php artisan innodite:migrate-plan --manifest=acme.order.json

# Tenant "beta-corp"
php artisan innodite:migrate-plan --manifest=beta-corp.order.json
Example 3 — Preview without executing (dry-run)
bash
# Safe preview — nothing is touched in the database
php artisan innodite:migrate-plan --manifest=acme.order.json --dry-run

# Output example:
# [DRY-RUN] Would migrate: User/2024_01_01_create_users_table.php (acme_db)
# [DRY-RUN] Would migrate: Invoice/2024_01_10_create_invoices_table.php (acme_db)
# [DRY-RUN] 2 migrations pending. Not executed (dry-run mode).
Example 4 — Migrate AND seed in one command
bash
# Useful for fresh environments, CI/CD pipelines
php artisan innodite:migrate-plan --manifest=central.order.json --seed

# With dry-run to preview both migrations and seeders
php artisan innodite:migrate-plan --manifest=central.order.json --seed --dry-run
Example 5 — Batch: run all tenants
bash
# Shell script for all tenant manifests
for manifest in module-maker-config/migrations/*.order.json; do
  filename=$(basename "$manifest")
  echo "==> Migrating: $filename"
  php artisan innodite:migrate-plan --manifest="$filename"
done

Manifest File Format

Manifests are stored in module-maker-config/migrations/:

jsonmodule-maker-config/migrations/central.order.json
{
  "migrations": [
    "User:Central/2024_01_01_000000_create_central_users_table.php",
    "Role:Central/2024_01_02_000000_create_central_roles_table.php",
    "Permission:Central/2024_01_03_000000_create_central_permissions_table.php"
  ],
  "seeders": [
    "User:Central/CentralUserDatabaseSeeder",
    "Role:Central/CentralRoleDatabaseSeeder"
  ]
}
🛡️ Guard Rail

Before any migration runs, the Guard Rail validates that the connection_key from contexts.json exists in config/database.php. If it doesn't, a ConnectionNotConfiguredException is thrown and nothing executes. --dry-run skips this check.

🎯
php artisan innodite:migrate-one {coordinate} [options]

Runs a single migration identified by its coordinate string. Automatically detects which manifest(s) the migration belongs to, validates the DB connection, registers the coordinate if missing, and executes the migration on the correct connection.

migrations targeted

Signature

bash
php artisan innodite:migrate-one {coordinate}
    {--manifest=}   # force a specific manifest
    {--yes}         # skip confirmation prompts
    {--dry-run}     # preview without executing

Understanding the Coordinate Format

A coordinate uniquely identifies a migration file: ModuleName:ContextPath/filename.php

CoordinateResolves to
User:Central/2024_01_01_create_users_table.phpModules/User/Database/Migrations/Central/2024_01_01_create_users_table.php
Invoice:Tenant/Acme/2024_02_01_create_invoices_table.phpModules/Invoice/Database/Migrations/Tenant/Acme/2024_02_01_create_invoices_table.php
Order:Shared/2024_03_01_create_orders_table.phpModules/Order/Database/Migrations/Shared/2024_03_01_create_orders_table.php

Options

OptionDescription
coordinateRequired. The migration coordinate in ModuleName:ContextPath/filename.php format.
--manifest=Force a specific manifest. Without this, the command auto-detects the correct manifest(s).
--yesAuto-confirm all prompts (useful in scripts/CI).
--dry-runPreview what would happen without touching the DB.

Examples

Example 1 — Migrate a central table
bash
php artisan innodite:migrate-one "User:Central/2024_01_01_000000_create_central_users_table.php"
Example 2 — Migrate a tenant-specific table
bash
php artisan innodite:migrate-one "Invoice:Tenant/Acme/2024_02_01_000000_create_acme_invoices_table.php"
Example 3 — Dry-run before executing
bash
php artisan innodite:migrate-one "Product:Central/2024_03_01_000000_create_central_products_table.php" --dry-run
Example 4 — Force a specific manifest and skip prompt
bash
php artisan innodite:migrate-one \
  "Order:Tenant/BetaCorp/2024_04_01_create_beta_orders_table.php" \
  --manifest=beta-corp.order.json \
  --yes
Example 5 — Migration exists in multiple manifests (interactive)
bash
# When a coordinate maps to multiple manifests (e.g. Shared context),
# the command shows an interactive selector:
php artisan innodite:migrate-one "Notification:Shared/2024_05_01_create_notifications_table.php"

# Output:
# This migration appears in multiple manifests. Which one to use?
# [0] central.order.json (central)
# [1] acme.order.json (acme)
# [2] beta-corp.order.json (beta-corp)
# > _
💡 Auto-register

If the coordinate isn't already in the manifest, migrate-one automatically registers it before executing. This keeps your manifests in sync without needing migration-sync.

🔄
php artisan innodite:migration-sync [options]

Scans all Modules/*/Database/Migrations/ and Modules/*/Database/Seeders/ directories, detects files and classes not yet registered in any manifest, and adds them automatically. Keeps your order files in perfect sync with your codebase.

sync manifests

Signature

bash
php artisan innodite:migration-sync
    {--manifest=}       # sync only one specific manifest
    {--all-manifests}   # sync all manifests (central + all tenants)
    {--yes}             # skip confirmation prompts
    {--dry-run}         # preview changes without writing

Options

OptionDescription
--manifest=Sync only the specified manifest (e.g. acme.order.json).
--all-manifestsScan and sync all manifests: central.order.json + one per tenant.
--yesSkip confirmation — useful in automation.
--dry-runShow what would be added without modifying files.

Examples

Example 1 — Sync all manifests (recommended after make-module)
bash
# After generating new modules, auto-register everything
php artisan innodite:migration-sync --all-manifests

# Output:
# Syncing central.order.json...
#   + User:Central/2024_01_01_000000_create_central_users_table.php
#   + User:Central/CentralUserDatabaseSeeder
# Syncing acme.order.json...
#   + Invoice:Tenant/Acme/2024_02_01_000000_create_acme_invoices_table.php
# Done. 3 coordinates added across 2 manifests.
Example 2 — Sync only the central manifest
bash
php artisan innodite:migration-sync --manifest=central.order.json
Example 3 — Preview before committing (dry-run)
bash
php artisan innodite:migration-sync --all-manifests --dry-run

# Shows every coordinate that would be added
# without writing to any .order.json file
Example 4 — CI/CD: sync after deploy
bash
# Recommended deploy script
php artisan innodite:migration-sync --all-manifests --yes
php artisan innodite:migrate-plan --manifest=central.order.json
php artisan innodite:migrate-plan --manifest=acme.order.json
Example 5 — After adding a migration manually
bash
# You manually created a migration in Modules/Billing/Database/Migrations/Central/
# Register it into the manifest without running it yet
php artisan innodite:migration-sync --manifest=central.order.json --yes

# Then review the manifest before running
cat module-maker-config/migrations/central.order.json
💡 Best Practice

Run innodite:migration-sync --all-manifests right after every innodite:make-module call. This ensures your manifests always reflect the current state of your Modules/ directory.

✅
php artisan innodite:check-env

Audits your Laravel application's environment to ensure all components required by Module Maker are properly configured. Checks permissions, Inertia sharing, and middleware registration, then provides copy-paste code snippets for any missing setup.

diagnostics environment

What it checks

1
User model permissions support

Verifies that your App\Models\User implements either Spatie's HasPermissions or the InnoditeUserPermissions interface.

2
HandleInertiaRequests middleware

Checks that auth.permissions is shared in the HandleInertiaRequests::share() method.

3
InnoditeContextBridge middleware

Verifies the context bridge middleware is registered in the web middleware group.

Examples

Example 1 — Run environment check
bash
php artisan innodite:check-env

# Example output (all passing):
# ✅ User model implements HasPermissions (Spatie)
# ✅ HandleInertiaRequests shares auth.permissions
# ✅ InnoditeContextBridge is registered in web group

# Example output (with issues):
# ❌ User model does not implement permissions interface
#    → Add HasPermissions trait from Spatie, OR implement InnoditeUserPermissions
#
#    // Option A: Spatie
#    use Spatie\Permission\Traits\HasPermissions;
#    class User extends Authenticatable { use HasPermissions; }
#
#    // Option B: Custom interface
#    use Innodite\LaravelModuleMaker\Contracts\InnoditeUserPermissions;
#    class User extends Authenticatable implements InnoditeUserPermissions {
#        public function getInnoditePermissions(): array { return [...]; }
#    }
Fix: Adding InnoditeUserPermissions to User model
phpapp/Models/User.php
use Innodite\LaravelModuleMaker\Contracts\InnoditeUserPermissions;

class User extends Authenticatable implements InnoditeUserPermissions
{
    public function getInnoditePermissions(): array
    {
        // Your custom permission resolution:
        return $this->roles
            ->flatMap(fn($r) => $r->permissions)
            ->pluck('name')
            ->unique()
            ->values()
            ->toArray();
    }
}

InnoditeContextBridge Middleware

The context bridge middleware automatically detects the active context from the current request and shares it — along with the user's permissions — to your Inertia frontend on every request.

What it shares with Inertia

javascriptAvailable in all Vue pages via usePage().props.auth
// Inertia shared props
{
  auth: {
    context: {
      route_prefix: "acme",        // matches the active context
      permission_prefix: "acme"    // for building permission strings
    },
    permissions: [
      "acme_users_index",
      "acme_invoices_store",
      "acme_invoices_destroy"
      // ...all permissions for the current user
    ]
  }
}

Context Detection Strategy

The middleware compares the current request against ALL contexts in priority order:

  1. Route name prefix — matches route_name from contexts.json
  2. URL path prefix — matches route_prefix from contexts.json

Using context data in Vue components

javascriptresources/js/Pages/Acme/Invoice/Index.vue
import { usePage, computed } from '@inertiajs/vue3'

const page = usePage()

const context = computed(() => page.props.auth.context)
const permissions = computed(() => page.props.auth.permissions)

// Check permission dynamically using context prefix
const canCreate = computed(() =>
  permissions.value.includes(`${context.value.permission_prefix}_invoices_store`)
)

// Build route names dynamically
const indexRoute = computed(() =>
  `${context.value.route_prefix}.invoices.index`
)

ContextResolver — Programmatic Access

Use ContextResolver anywhere in your code to access context configuration:

php
use Innodite\LaravelModuleMaker\Support\ContextResolver;

// Find a context by ID (searches central, shared, AND all tenants)
$context = ContextResolver::find('acme');
// ['id' => 'acme', 'connection_key' => 'acme_db', ...]

// Get only the central context (object lookup)
$central = ContextResolver::resolve('central');

// Get all tenants as a flat array
$tenants = ContextResolver::allTenants();

// Get ALL contexts as a flat array (including central, shared, etc.)
$all = ContextResolver::allItems();

// Validate that a connection exists in config/database.php
ContextResolver::validateConnection('acme');  // throws if not found
